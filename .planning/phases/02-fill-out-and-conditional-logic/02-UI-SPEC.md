## Spacing Scale

Declared values (inherited from Phase 1, unchanged):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Chip gaps, badge padding, icon gaps |
| sm | 8px | Card margin-bottom, row gap between items |
| md | 12px | Default row padding vertical, body horizontal padding, section top padding |
| md+ | 16px | Row padding horizontal, card inner padding horizontal |
| lg | 24px | Page-level section breaks |
| xl | 32px | Empty-state padding vertical |

Exceptions:
- Touch targets: minimum 44px tall for all tappable field rows (checkbox/yes-no row tap, entire row). Enforced via `min-height: 44px` on `.fill-row`.
- Progress counter area: 12px padding all sides inside `.fill-progress` card.
- Corrective action card: 12px top/bottom padding, 16px left/right padding — matches `.field-settings` pattern.
- Severity pills: 8px vertical × 16px horizontal padding inside each pill.

Source: Phase 1 `01-UI-SPEC.md` spacing scale; 12px extension retained.

Accent (`var(--info-bg)` / `var(--info-tx)`) reserved for:
1. Active tab indicator (`.tabs button.on`) — inherited from Phase 1
2. "Submit checklist" primary CTA button background
3. Focus ring on text/temperature inputs (border-color: `var(--info-tx)`)
4. Back link color ("‹ Checklists")
5. Progress bar fill (`.fill-progress-bar-fill`)
6. Active severity pill background when selected

Accent is NOT used for: general field rows, section headers, completed checkmarks (use `var(--txt)`), the corrective action card, or decorative elements.

New semantic color — fail card:
- `.fail-card` background: `rgba(192,57,43,0.08)` light / `rgba(192,57,43,0.15)` dark
- `.fail-card` label color: `#c0392b`
- This is not a new CSS variable — use inline class CSS.

Source: Phase 1 `01-UI-SPEC.md` color contract + CONTEXT.md D-04 (red-tinted fail card).

## Interaction Contracts

### Checklist List View

- **Entry point:** Fill Out tab (#s1), default tab on page load.
- **Day filter:** On render, compute `today = new Date().getDay()`. Only show templates where `template.active_days.includes(today)` OR `template.active_days.length === 0` (no restriction). Templates that do not match are not rendered — no "hidden" mention.
- **Card contents:** Template name at 15px weight 500; section count at 12px `var(--mut)` (e.g., "3 sections"); progress bar showing completion percentage (0% on first open); pill showing "Requires approval" in `var(--warn-bg)` / `var(--warn-tx)` if `template.requires_approval === true`.
- **Tap to open:** Tapping anywhere on a checklist card enters the runner view for that template.
- **Empty state:** Rendered when no templates match today's day filter. See Copywriting Contract.

### Checklist Runner — Opening

- **Sub-view state:** `state.fillView` = `'list'` or `'runner'`. `state.activeResponseId` points to the current response object.
- **On open:** Create a new response object in `state.responses[templateId]` — `{ templateId, startedAt, completedItems: {}, failNotes: {} }`. If a response already exists for this template today, resume it (answers preserved).
- **Back navigation:** "‹ Checklists" link at top — calls `backToFillList()`, does NOT clear response state (partial answers persist).

### Checklist Runner — Section and Field Rendering

- **Sections:** Rendered in `section.order` sequence. Section header uses `.cat` class (sticky). Sections hidden by day-of-week condition are skipped entirely.
- **Field visibility:** Before rendering each field, evaluate `field.condition` using the same JsonLogic/condition logic from Phase 1. Hidden fields are not rendered and their answer is cleared from `state.responses[templateId].completedItems[fieldId]`.
- **Field rendering by type:**
  - `checkbox`: Full-row tap target (`.fill-row`). Left: label at 14px. Right: empty circle (uncompleted) or green ✓ + `.attr-badge` (completed).
  - `yes_no`: Same as checkbox row. Below the row: two tappable answer pills "Yes" / "No" at 14px — OR render as two buttons side by side. Tap "No" triggers fail card.
  - `text`: Label at 14px above, `<input type="text">` inline below. Tap to focus. No expand/collapse.
  - `temperature`: Label at 14px above, `<input type="number">` + `°F` suffix inline. Tap to focus. On blur or input event, evaluate against `field.fail_trigger` thresholds.

### Checkbox / Yes-No Completion

- **Trigger:** Tap anywhere on `.fill-row` (full row).
- **Behavior on tap:** Toggle `completedItems[fieldId]` between null and `{ answeredBy: CURRENT_USER, answeredAt: new Date().toISOString(), value: true }`. Re-render row.
- **Completed state:** Label gets `text-decoration: line-through` and `color: var(--mut)`. Green ✓ character (`✓`) appears to the right using `color: var(--txt)` (not accent). Attribution badge appears beside the checkmark: `"JM · 2:15p"` at 12px `var(--mut)`.
- **Animation:** None — instant state change + re-render (matches codebase convention).

### Yes-No "No" Fail Trigger

- **Trigger:** User taps "No" answer pill on a `yes_no` field.
- **Behavior:** Record answer as `{ value: false, ... }`. Immediately render `.fail-card` below the field row without blocking other fields.
- **Dismissal:** Fail card stays visible until the answer is changed to "Yes" (card removed) or checklist is submitted.

### Temperature Out-of-Range Fail Trigger

- **Trigger:** On `input` event of the temperature `<input>`, evaluate `value < field.fail_trigger.min || value > field.fail_trigger.max` (only evaluate if both min and max are set and value is not empty).
- **Behavior:** If out of range, render `.fail-card` below the field row. If value is then corrected to within range, remove the fail card immediately.
- **Out-of-range indicator:** Small amber label below the input (`var(--warn-tx)`) showing "Out of range (350–500°F)" before the fail card.

### Corrective Action Card

- **Structure:** `.fail-card` card with red-tinted background (`rgba(192,57,43,0.08)`).
  1. Header label: "Corrective action required" at 12px weight 500 `#c0392b`.
  2. Text note input: `<textarea>` with placeholder "Describe what you did…", 14px, 3 rows, full width. Follows `.field input` border/background style.
  3. Photo stub: `.photo-stub-btn` — a camera emoji button (📷) at 14px with label "Add photo" and sub-label "Coming in Phase 3" at 12px `var(--mut)`. Non-functional — no click handler.
  4. Severity picker: `.severity-pills` row — three `.severity-pill` buttons: "Minor" / "Major" / "Critical". Inactive: `var(--brd)` border, transparent background. Active: `var(--info-bg)` background, `var(--info-tx)` text.
- **State:** `state.responses[templateId].failNotes[fieldId]` stores `{ note, severity }`.
- **Not blocking:** Fail card does not prevent the user from completing other rows.

### Progress Tracking

- **Counter:** Rendered at top of runner view inside a small card: "X of Y items complete" at 12px `var(--mut)`. Y counts only visible (not hidden by conditions) fields. X counts fields with a truthy value in `completedItems`.
- **Progress bar on list card:** `width: {X/Y * 100}%` on `.fill-progress-bar-fill`. Updates when the user returns to list view.
- **Re-compute:** On every state mutation that changes `completedItems`, recompute X and re-render the counter.

### Submit Checklist

- **Trigger:** "Submit checklist" `.btn-primary` button at the bottom of the runner view, always visible (not gated on completion).
- **Behavior (mock only):** Show an inline confirmation message below the button: "Checklist submitted. Thanks, JM!" at 14px `var(--mut)`. Button becomes disabled (opacity 0.5, `pointer-events: none`). No navigation change.
- **No modal:** Confirmation is inline text, not a modal or alert. Matches the codebase's no-modal convention.

### Conditional Logic (Skip Logic and Day-of-Week)

- **Day-of-week filtering:** Evaluated once on render of the runner view. Sections with a `condition.days` array that does not include today are skipped.
- **Skip logic:** Evaluated on every state change. For each field with a `condition` object, evaluate the condition against `state.responses[templateId].completedItems`. Hidden fields are removed from the DOM and their `completedItems` entry is deleted.
- **Pure function:** Condition evaluation is a pure function — `isFieldVisible(field, completedItems) → boolean`. No DOM reads.

### Event Delegation

- ONE `click` listener on `#fillout-body`.
- ONE `input` listener on `#fillout-body`.
- Use `event.target.closest('[data-action]')` to route events.
- Do NOT attach listeners inside render functions.
