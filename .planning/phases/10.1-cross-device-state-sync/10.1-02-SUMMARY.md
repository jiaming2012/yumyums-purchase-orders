# Phase 10.1 Plan 02: WebSocket Hub, LISTEN/NOTIFY Listener, and Catch-up Endpoint Summary

**Real-time delivery pipeline: pgxlisten on ops_channel fans out to per-user WebSocket clients via channel-based hub, with /ops/since catch-up for reconnecting devices**

## Accomplishments

- Hub with channel-based concurrency (no mutex): register/unregister/broadcast select loop, per-user client slices, slow-client detection
- pgxlisten listener on ops_channel: fetches full Op via GetOpByID, resolves target users via ResolveEntityAccess, fans out to hub.broadcast
- WsHandler: upgrades HTTP to WebSocket, registers Client in hub, runs write pump goroutine + read loop, defers unregister on disconnect
- OpsSinceHandler: parses lamport_ts query param, delegates to OpsSince, returns JSON array
- Routes wired in main.go: /ws at top-level with auth middleware, /ops/since inside /api/v1/workflow

## Decisions Made

- pgconn import is `github.com/jackc/pgx/v5/pgconn` — the standalone `github.com/jackc/pgconn` does not exist in pgx v5. Fixed in Task 1 as Rule 3 (blocking import error).
- Fan-out fallback: when `ResolveEntityAccess` returns empty (e.g. entity not yet in template_assignments), listener falls back to `op.UserID` so the author always receives their own op.
- `/ws` mounted outside `/api/v1` prefix in a dedicated `r.Group` with auth middleware — avoids chi route conflict and matches plan spec.
