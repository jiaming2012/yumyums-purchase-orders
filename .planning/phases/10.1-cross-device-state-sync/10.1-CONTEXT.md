# Phase 10.1: Cross-Device State Sync - Context

**Gathered:** 2026-04-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Real-time state synchronization across devices and users. When any user mutates state (field response, submission, approval, template edit), all other connected clients with access to that entity see the change in real-time via WebSocket. Offline devices catch up on resume via `GET /ops/since?lamport_ts=N`. Conflicts resolved by LWW with Lamport timestamps + device_id tiebreaker.

This phase adds the sync infrastructure (ops table, WebSocket hub, Lamport clocks, catch-up endpoint, conflict UX). It does NOT rewrite existing write paths — `autoSaveField`/`saveResponse` stays, but the server emits ops after each write. Future phases migrate each module to `POST /ops` as they get backend work.

</domain>

<decisions>
## Implementation Decisions

### Migration Strategy
- **D-01:** Incremental migration. Add ops table + WebSocket infra now. Existing write paths (`POST /saveResponse`, `POST /submitChecklist`, etc.) remain but server-side handlers also INSERT into ops and fire `pg_notify` after each write. Future phases (11, 12) migrate their modules to `POST /ops` as a universal write endpoint.
- **D-02:** No frontend write-path changes in this phase. `autoSaveField`, `submitChecklistToAPI`, approval handlers keep working as-is. The op-log is populated server-side, not client-side, during this transitional phase.

### Op-Log Scope
- **D-03:** Universal op log — ALL mutations go through the ops table. Entity types: field responses (SET_FIELD), submissions (SUBMIT_CHECKLIST), approvals (APPROVE_ITEM, REJECT_ITEM), template edits (SAVE_TEMPLATE), template archive (ARCHIVE_TEMPLATE). This creates a complete audit trail from day one.
- **D-04:** Op log schema per PDF: `ops(id UUID, device_id TEXT, user_id UUID, entity_id UUID, entity_type TEXT, op_type TEXT, payload JSONB, lamport_ts BIGINT, server_ts TIMESTAMPTZ, applied BOOLEAN)`. Indexes on `(user_id, lamport_ts)` and `(entity_id)`.
- **D-05:** Entity tables (submission_responses, submissions, templates, etc.) get a `lamport_ts BIGINT` column for LWW guard. Atomic writes: INSERT ops + UPDATE entity in one transaction.

### Real-Time Transport
- **D-06:** WebSocket for real-time push. Go server listens on Postgres `LISTEN ops_channel`, fans out to WebSocket hub (`map[userID][]*websocket.Conn`). Client connects via `ws://host/ws` (or `wss://` over Tailscale).
- **D-07:** Reconnect-on-visibility pattern for iOS Safari background kills. `visibilitychange` event checks `ws.readyState`; if closed, reconnect + catch-up via `GET /ops/since?lamport_ts=N`. No polling loop.
- **D-08:** Catch-up endpoint: `GET /api/v1/ops/since?lamport_ts=N` returns all ops with `lamport_ts > N` for the authenticated user's accessible entities. Used on WS reconnect and initial page load.

### Sync Scope: Cross-User, Not Just Cross-Device
- **D-09:** WebSocket fan-out targets ALL users with access to the entity, not just the same user on multiple devices. If Maria checks a field on a shared checklist, Jamal's phone sees it live. The hub must resolve entity access (e.g., template assignees, approvers) to determine which connected users receive each op.

### Conflict Resolution
- **D-10:** LWW with Lamport timestamps. Higher `lamport_ts` wins. Ties broken by lexicographically higher `device_id` — deterministic, no coin flip.
- **D-11:** Each device maintains a local `lamport_ts` counter in IndexedDB. Increments on every write, updates to `max(local, received) + 1` on every received op.
- **D-12:** Server returns 409 with winner state on conflict. Losing device applies winner state directly.

### Conflict UX
- **D-13:** Inline field highlight (blue flash, fades after 2s) on fields changed by other users/devices. Grouped toast notification: "2 fields updated by Maria". This is a REUSABLE PATTERN — designed to work across all tools (workflows, onboarding, inventory), not just this phase.
- **D-14:** On 409 conflict, field reverts to winner value with the same blue highlight + toast explaining what happened. No modal, no blocking UI.

### Offline Behavior
- **D-15:** Existing IndexedDB `submitQueue` + `drainQueue()` on `online` event stays for offline submissions. Op-log catch-up happens after drain completes — reconnect WS, then `GET /ops/since?lamport_ts=N` to pull missed ops from other users/devices.
- **D-16:** Pending ops from other users that arrived while offline are applied in lamport_ts order during catch-up. UI shows blue highlights for all caught-up changes.

### Claude's Discretion
- WebSocket library choice for Go (gorilla/websocket vs nhooyr.io/websocket)
- Client-side WebSocket reconnect backoff strategy (exponential with jitter)
- Device ID generation and storage approach
- Blue highlight CSS implementation (animation, timing, color in dark mode)
- Toast grouping logic (batch window, max items shown)
- Hub data structure for resolving entity access → connected users
- Whether `device_id` column goes on existing tables or only on ops

</decisions>

<specifics>
## Specific Ideas

- **PDF architecture is the blueprint** — follow the op log + materialized tables + LWW + WebSocket pattern from `pwa-sync-architecture.pdf`. Adapt entity types from generic "todos" to Yumyums entities (field responses, submissions, approvals, templates).
- **Blue highlight is a shared component** — the sync highlight + grouped toast pattern must be designed as a reusable module, not hardcoded into workflows.html. Future tools (onboarding, inventory, users) will use the same pattern.
- **Cross-user sync is first-class** — this is not just "my phone and my tablet." Different crew members working the same checklist must see each other's changes live. The WS hub resolves entity access, not just user identity.

</specifics>
