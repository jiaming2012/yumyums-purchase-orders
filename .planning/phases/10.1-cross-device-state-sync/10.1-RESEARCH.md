### Locked Decisions

- **D-01:** Incremental migration. Add ops table + WebSocket infra now. Existing write paths (`POST /saveResponse`, `POST /submitChecklist`, etc.) remain — server-side handlers also INSERT into ops and fire `pg_notify` after each write.
- **D-02:** No frontend write-path changes in this phase. `autoSaveField`, `submitChecklistToAPI`, approval handlers keep working as-is.
- **D-03:** Universal op log — ALL mutations: SET_FIELD, SUBMIT_CHECKLIST, APPROVE_ITEM, REJECT_ITEM, SAVE_TEMPLATE, ARCHIVE_TEMPLATE.
- **D-04:** Op log schema: `ops(id UUID, device_id TEXT, user_id UUID, entity_id UUID, entity_type TEXT, op_type TEXT, payload JSONB, lamport_ts BIGINT, server_ts TIMESTAMPTZ, applied BOOLEAN)`. Indexes on `(user_id, lamport_ts)` and `(entity_id)`.
- **D-05:** Entity tables get a `lamport_ts BIGINT` column for LWW guard. Atomic writes: INSERT ops + UPDATE entity in one transaction.
- **D-06:** WebSocket for real-time push. Go server listens on Postgres `LISTEN ops_channel`, fans out to WebSocket hub (`map[userID][]*websocket.Conn`). Client connects via `wss://host/ws`.
- **D-07:** Reconnect-on-visibility pattern for iOS Safari. `visibilitychange` event checks `ws.readyState`; if closed, reconnect + catch-up via `GET /ops/since?lamport_ts=N`.
- **D-08:** Catch-up endpoint: `GET /api/v1/ops/since?lamport_ts=N` returns all ops with `lamport_ts > N` for the authenticated user's accessible entities.
- **D-09:** WebSocket fan-out targets ALL users with access to the entity (not just same-user multi-device). Hub resolves entity access.
- **D-10:** LWW with Lamport timestamps. Higher `lamport_ts` wins. Ties broken by lexicographically higher `device_id`.
- **D-11:** Each device maintains a local `lamport_ts` counter in IndexedDB. Updates to `max(local, received) + 1` on every received op.
- **D-12:** Server returns 409 with winner state on conflict. Losing device applies winner state directly.
- **D-13:** Inline field highlight (blue flash, fades after 2s) on fields changed by other users/devices. Grouped toast: "2 fields updated by Maria". Reusable pattern for all tools.
- **D-14:** On 409 conflict, field reverts to winner value with blue highlight + toast.
- **D-15:** Existing IndexedDB `submitQueue` + `drainQueue()` on `online` event stays. Op-log catch-up happens after drain completes.
- **D-16:** Pending ops from other users that arrived while offline are applied in lamport_ts order during catch-up.

## Summary

Phase 10.1 adds real-time cross-user/cross-device synchronization to the Yumyums HQ PWA. The architecture is an **op-log with materialized entity tables**: every mutation inserts an immutable row into `ops`, entity tables are updated atomically in the same transaction, and Postgres LISTEN/NOTIFY fans out to a WebSocket hub that delivers ops to all connected users with access to the affected entity.

The chosen stack (coder/websocket v1.8.14 + pgxlisten for the Go server, vanilla JS + IndexedDB for the client) integrates cleanly with the existing chi router + pgx pool + IndexedDB offline queue already in the codebase. The incremental migration strategy means no existing write path changes in this phase — handlers gain a post-write `emitOp()` helper call that fires as a goroutine.

The most complex piece is the **WebSocket hub**: it must maintain a `map[userID][]Conn` structure and, on receiving a NOTIFY from Postgres, resolve which userIDs have access to the affected entity before fanning out. For Phase 10.1, the entity access query can be simple: look up `template_assignments` to find assignees and approvers for the affected template.

**Primary recommendation:** Use `coder/websocket v1.8.14` (actively maintained successor to gorilla), `pgxlisten` for reconnecting LISTEN/NOTIFY, and a channel-per-connection hub pattern with a single goroutine owning the client map.

### Pattern 1: WebSocket Hub (Go)

**What:** A single goroutine owns the client map. All register/unregister/broadcast operations go through channels to avoid mutex.

**When to use:** Any time multiple goroutines need to fan-out to a variable set of WebSocket connections.

```go
// Source: gorilla/websocket chat example (canonical hub pattern) + coder/websocket API
type Client struct {
    userID string
    conn   *websocket.Conn
    send   chan []byte
}

type Hub struct {
    clients    map[string][]*Client // userID → connections
    register   chan *Client
    unregister chan *Client
    broadcast  chan BroadcastMsg
}

type BroadcastMsg struct {
    userIDs []string  // resolved from entity access query
    data    []byte
}

func (h *Hub) Run() {
    for {
        select {
        case c := <-h.register:
            h.clients[c.userID] = append(h.clients[c.userID], c)
        case c := <-h.unregister:
            // remove c from h.clients[c.userID]
        case msg := <-h.broadcast:
            for _, uid := range msg.userIDs {
                for _, c := range h.clients[uid] {
                    select {
                    case c.send <- msg.data:
                    default:
                        // buffer full — drop and close stale connection
                        close(c.send)
                    }
                }
            }
        }
    }
}
```

**coder/websocket upgrade in chi handler:**

```go
// Source: coder/websocket v1.8.14 README

        }
    }
}
```

### Pattern 2: Postgres LISTEN/NOTIFY via pgxlisten

**What:** A dedicated goroutine holds a single Postgres connection for LISTEN. On each NOTIFY, it parses the payload and calls `hub.Broadcast()` with the resolved userIDs.

**Key constraint:** NOTIFY payload is limited to **8000 bytes** (Postgres hard limit). The payload must contain only IDs and essential metadata — never full records.

```go
// Source: pkg.go.dev/github.com/jackc/pgxlisten
import "github.com/jackc/pgxlisten"

func StartListener(ctx context.Context, connStr string, hub *Hub, pool *pgxpool.Pool) {
    listener := &pgxlisten.Listener{
        Connect: func(ctx context.Context) (*pgx.Conn, error) {
            return pgx.Connect(ctx, connStr)
        },
        ReconnectDelay: 5 * time.Second,
    }

    listener.Handle("ops_channel", pgxlisten.HandlerFunc(func(
        ctx context.Context,
        notification *pgconn.Notification,
        conn *pgx.Conn,
    ) error {
        // payload: {"op_id":"uuid","entity_id":"uuid","entity_type":"field_response","user_id":"uuid"}
        var notif struct {
            OpID       string `json:"op_id"`
            EntityID   string `json:"entity_id"`
            EntityType string `json:"entity_type"`
            UserID     string `json:"user_id"`  // who made the change
        }
        if err := json.Unmarshal([]byte(notification.Payload), &notif); err != nil {
            return nil // malformed — skip
        }

        // Fetch full op row from DB (payload is just IDs)
        op, err := getOpByID(ctx, pool, notif.OpID)
        if err != nil {
            return nil
        }

        // Resolve which users have access to this entity
        userIDs, err := resolveEntityAccess(ctx, pool, notif.EntityID, notif.EntityType)
        if err != nil {
            return nil
        }

        data, _ := json.Marshal(op)
        hub.broadcast <- BroadcastMsg{userIDs: userIDs, data: data}
        return nil
    }))

    go func() {
        if err := listener.Listen(ctx); err != nil {
            log.Printf("pgxlisten stopped: %v", err)
        }
    }()
}
```

### Pattern 3: emitOp() Helper in Existing Handlers

**What:** Post-write fire-and-forget goroutine (same pattern as `cleanupOldDrafts`). Existing handlers call it after their primary write succeeds.

```go
// Source: existing cleanupOldDrafts pattern in MyChecklistsHandler
func emitOp(pool *pgxpool.Pool, op OpInput) {
    go func() {
        ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
        defer cancel()
        if err := insertOpAndNotify(ctx, pool, op); err != nil {
            log.Printf("emitOp error: %v", err)
        }
    }()
}

// insertOpAndNotify runs in a single transaction:
// 1. INSERT INTO ops (...)
// 2. SELECT pg_notify('ops_channel', payload)
func insertOpAndNotify(ctx context.Context, pool *pgxpool.Pool, op OpInput) error {
    tx, err := pool.Begin(ctx)
    if err != nil {
        return err
    }
    defer tx.Rollback(ctx)

    var opID string
    err = tx.QueryRow(ctx, `
        INSERT INTO ops (device_id, user_id, entity_id, entity_type, op_type, payload, lamport_ts)
        VALUES ($1, $2, $3, $4, $5, $6, $7)
        RETURNING id`,
        op.DeviceID, op.UserID, op.EntityID, op.EntityType, op.OpType, op.Payload, op.LamportTS,
    ).Scan(&opID)
    if err != nil {
        return err
    }

    notifPayload, _ := json.Marshal(map[string]string{
        "op_id": opID, "entity_id": op.EntityID,
        "entity_type": op.EntityType, "user_id": op.UserID,
    })
    _, err = tx.Exec(ctx, "SELECT pg_notify('ops_channel', $1)", string(notifPayload))
    if err != nil {
        return err
    }
    return tx.Commit(ctx)
}
```

**Where to add `emitOp()` calls:**

| Handler | After which write succeeds | Op type |
|---------|--------------------------|---------|
| `SaveResponseHandler` | `saveResponse()` returns nil | `SET_FIELD` |
| `SubmitChecklistHandler` | `submitChecklist()` returns nil | `SUBMIT_CHECKLIST` |

### Pattern 4: Lamport Clock + Device ID (Vanilla JS)

**What:** IndexedDB-backed clock. Increments on every local write; updates to `max(local, received) + 1` on every received op.

```javascript
// Stored in IndexedDB 'hq_offline_v1' → store 'syncMeta', key 'clock'
// { lamport_ts: Number, device_id: String }

class LamportClock {
  constructor(db) { this._db = db; this._ts = 0; this._deviceId = null; }

  static async init(db) {
    const clock = new LamportClock(db);
    const meta = await idbGet(db, 'syncMeta', 'clock');
    if (meta) {
      clock._ts = meta.lamport_ts;
      clock._deviceId = meta.device_id;
    } else {
      clock._deviceId = crypto.randomUUID();
      await idbPut(db, 'syncMeta', { id: 'clock', lamport_ts: 0, device_id: clock._deviceId });
    }
    return clock;
  }

  // Call before every local write — returns incremented ts
  async tick() {
    this._ts += 1;
    await idbPut(this._db, 'syncMeta', { id: 'clock', lamport_ts: this._ts, device_id: this._deviceId });
    return this._ts;
  }

  // Call on every received op from server
  async receive(remoteTs) {
    this._ts = Math.max(this._ts, remoteTs) + 1;
    await idbPut(this._db, 'syncMeta', { id: 'clock', lamport_ts: this._ts, device_id: this._deviceId });
  }

  get ts() { return this._ts; }
  get deviceId() { return this._deviceId; }
}
```

**Note:** The IndexedDB setup requires adding a `syncMeta` object store (key `id`) to the DB upgrade in `getDB()`. The current `hq_offline_v1` version is `1` — bump to version `2` in `indexedDB.open(HQ_DB, 2)` and add the store in `onupgradeneeded`.

### Pattern 5: WebSocket Client (Vanilla JS)

**What:** Exponential backoff with jitter. `visibilitychange` triggers reconnect + catch-up. No polling.

```javascript
const WS_BASE = location.protocol === 'https:' ? 'wss://' : 'ws://';
let _ws = null, _wsBackoff = 500, _wsAttempts = 0;
const WS_MAX_BACKOFF = 30000;
const WS_MAX_ATTEMPTS = 15;

function wsConnect() {
  if (_wsAttempts >= WS_MAX_ATTEMPTS) return; // surface error to user instead
  _ws = new WebSocket(WS_BASE + location.host + '/ws');

  _ws.onopen = () => {
    _wsBackoff = 500;
    _wsAttempts = 0;
    // Catch up on any missed ops
    wsCatchUp();
  };

  _ws.onmessage = (evt) => {
    const op = JSON.parse(evt.data);
    LAMPORT_CLOCK.receive(op.lamport_ts);
    applyOp(op);
  };

  _ws.onclose = () => {
    const jitter = Math.random() * 1000;
    const delay = Math.min(_wsBackoff + jitter, WS_MAX_BACKOFF);
    _wsBackoff = Math.min(_wsBackoff * 2, WS_MAX_BACKOFF);
    _wsAttempts++;
    setTimeout(wsConnect, delay);
  };
}

// iOS Safari: reconnect when page becomes visible
document.addEventListener('visibilitychange', () => {
  if (document.visibilityState === 'visible') {
    if (!_ws || _ws.readyState === WebSocket.CLOSED || _ws.readyState === WebSocket.CLOSING) {
      _wsAttempts = 0; // reset backoff on intentional resume
      wsConnect();
    } else if (_ws.readyState === WebSocket.OPEN) {
      wsCatchUp(); // WS alive but may have missed ops while hidden
    }
  }
});

async function wsCatchUp() {
  const ts = LAMPORT_CLOCK.ts;
  const ops = await api('GET', '../ops/since?lamport_ts=' + ts, null);
  if (!Array.isArray(ops)) return;
  // Sort by lamport_ts ascending and apply in order
  ops.sort((a, b) => a.lamport_ts - b.lamport_ts);
  for (const op of ops) {
    await LAMPORT_CLOCK.receive(op.lamport_ts);
    applyOp(op);
  }
}
```

### Pattern 6: applyOp() — State-First Op Application

**What:** Incoming ops mutate JS state arrays first, then re-render. Follows the established state-first rendering pattern.

```javascript
function applyOp(op) {
  if (op.op_type === 'SET_FIELD') {
    const { field_id, value } = op.payload;
    // Skip if this op came from THIS device (already applied optimistically)
    if (op.device_id === LAMPORT_CLOCK.deviceId && op.lamport_ts <= LAMPORT_CLOCK.ts) return;

    FIELD_RESPONSES[field_id] = { value };
    // Update DRAFT_RESPONSES cache
    const existing = DRAFT_RESPONSES.find(d => d.field_id === field_id);
    if (existing) { existing.value = value; existing.answered_at = op.server_ts; }
    else DRAFT_RESPONSES.push({ field_id, value, answered_at: op.server_ts });

    // Re-render the field if it's currently visible
    renderFieldResponse(field_id);

    // Flash highlight + toast
    flashField(field_id);
    enqueueSyncToast(op.user_id, op.payload.user_name || 'Someone', [field_id]);
  }
  // ... other op_types: SUBMIT_CHECKLIST, APPROVE_ITEM, etc.
}
```

```javascript
function flashField(fieldId) {
  const row = document.querySelector(`[data-field-id="${fieldId}"]`);
  if (!row) return;
  row.classList.remove('sync-flash'); // reset if already flashing
  void row.offsetWidth;               // force reflow
  row.classList.add('sync-flash');
  setTimeout(() => row.classList.remove('sync-flash'), 2000);
}

// Toast grouping: batch updates within 500ms window
let _toastQueue = {};
let _toastTimer = null;
function enqueueSyncToast(userId, userName, fieldIds) {
  if (!_toastQueue[userId]) _toastQueue[userId] = { name: userName, count: 0 };
  _toastQueue[userId].count += fieldIds.length;
  clearTimeout(_toastTimer);
  _toastTimer = setTimeout(flushSyncToast, 500);
}
function flushSyncToast() {
  const entries = Object.values(_toastQueue);
  if (!entries.length) return;
  // Render one toast: "3 fields updated by Maria, 1 by Jamal"
  const msg = entries.map(e => `${e.count} field${e.count>1?'s':''} updated by ${e.name}`).join(', ');
  showToast(msg);
  _toastQueue = {};
}
```

### Anti-Patterns to Avoid

- **Sending full records in pg_notify payload:** The 8000-byte limit is hard. Send only IDs; fetch the full op row from DB in the listener handler.
- **Using pgxpool for LISTEN:** `pgxpool` recycles connections — a pooled connection in LISTEN state will receive notifications while also being used for other queries, causing data corruption. Use a dedicated `pgx.Conn` (managed by pgxlisten).
- **Writing from multiple goroutines to gorilla/websocket:** This causes panics. Use `coder/websocket` (concurrent writes are safe) or serialize through the `send` channel.
- **Polling instead of visibilitychange:** Polling drains iOS battery. The visibilitychange + readyState check is the correct pattern.
- **Re-applying self-ops:** When `op.device_id === LAMPORT_CLOCK.deviceId` and the op arrived via WebSocket (round-trip confirmation), the UI state is already correct from the optimistic write. Skip or no-op.
- **Mutating `device_id` on existing entity tables:** The decision (D-10/D-11) stores `device_id` only in the `ops` table and IndexedDB. Do not add it to `submission_responses`, `checklist_submissions`, etc.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Postgres LISTEN/NOTIFY with reconnect | Custom `WaitForNotification` loop | `pgxlisten` | Connection drops cause silent notification loss; reconnect + backlog handling is subtle |
| WebSocket library for Go | Custom upgrade | `coder/websocket` | Concurrent write safety, context integration, active maintenance |
| Lamport clock persistence | Ad-hoc localStorage writes | IndexedDB with explicit transaction (pattern above) | localStorage is partitioned per iOS Safari PWA install, and sync across tabs isn't guaranteed |
| CSS flash animation | JS-driven style mutations | CSS `@keyframes` + class toggle | JS style mutations cause layout thrashing; keyframes are GPU-composited |

**Key insight:** The Postgres notification → WebSocket fan-out pipeline has multiple failure modes (connection drop, slow consumer, payload overflow) that are each independently subtle. Use `pgxlisten` to handle the Postgres side; use coder/websocket's buffered send channel pattern on the WS side.

## Entity Access Resolution

For WebSocket fan-out (D-09), the hub must answer: "which user IDs should receive this op?"

For Phase 10.1 scope, the access model is:
- `SET_FIELD` ops on a checklist → notify the `submitted_by` user + all approvers for the template
- `SUBMIT_CHECKLIST` → notify all approvers for the template
- `APPROVE_ITEM` / `REJECT_ITEM` → notify the `submitted_by` user
- `SAVE_TEMPLATE` / `ARCHIVE_TEMPLATE` → notify all users assigned to the template

Query pattern for `resolveEntityAccess`:

```go
// For SET_FIELD: entity_id = field_id, need to find template
// Better: pass template_id in op payload so we can query directly
func resolveEntityAccess(ctx context.Context, pool *pgxpool.Pool, templateID string) ([]string, error) {
    rows, err := pool.Query(ctx, `
        SELECT DISTINCT assignee_id::text
        FROM template_assignments
        WHERE template_id = $1 AND assignee_type = 'user'
    `, templateID)
    // ...collect and return
}
```

**Recommendation (Claude's discretion):** The `ops.payload` JSONB should always include `template_id` so the listener goroutine can resolve access with a single query. The NOTIFY payload (under 8000 bytes) should include `template_id` alongside `op_id`.

## Common Pitfalls

### Pitfall 1: IndexedDB Version Bump Required
**What goes wrong:** The `syncMeta` object store doesn't exist — `idbPut` to `syncMeta` fails silently or throws.
**Why it happens:** `indexedDB.open(HQ_DB, 1)` — the existing version is `1`. Adding a new object store requires bumping to `2` and handling `onupgradeneeded`.
**How to avoid:** Bump `HQ_DB` open version to `2` and add `createObjectStore('syncMeta', { keyPath: 'id' })` in `onupgradeneeded`. The existing `submitQueue` store must also be created in the version 1 case within the same handler.
**Warning signs:** `DOMException: IDBDatabase: The object store name 'syncMeta' does not exist.` in browser console.

### Pitfall 2: pg_notify Payload Overflow
**What goes wrong:** `SELECT pg_notify(...)` fails with `ERROR: payload string too long`.
**Why it happens:** Sending JSON with `payload` JSONB content in the notify. Limit is 8000 bytes.
**How to avoid:** Only include IDs in the NOTIFY payload: `op_id`, `entity_id`, `entity_type`, `user_id`, `template_id`. Never include `payload` JSONB.
**Warning signs:** Postgres error in server logs immediately after a write.

### Pitfall 3: Self-Op Re-Application Creates Visual Flicker
**What goes wrong:** User types in a field → autoSaveField fires → server returns 204 → server also emits op via WS → WS message arrives → `applyOp` flashes the field the user just typed in.
**Why it happens:** The WebSocket fan-out includes the originating user's own connections.
**How to avoid:** In `applyOp`, check `if (op.device_id === LAMPORT_CLOCK.deviceId) return;` to skip self-originated ops. The optimistic state is already correct.
**Warning signs:** Fields flash blue immediately after the user interacts with them.

### Pitfall 4: pgxpool Connection Used for LISTEN
**What goes wrong:** Notifications arrive inconsistently or not at all.
**Why it happens:** pgxpool returns connections back to the pool; if the LISTEN connection is returned, the next query on a different connection won't have the subscription.
**How to avoid:** Use `pgxlisten` which manages a dedicated connection outside the pool. Never call `LISTEN` on a pooled connection.
**Warning signs:** Sporadic missing notifications, especially after periods of low DB activity.

### Pitfall 5: iOS Safari Kills WS on Lock Screen
**What goes wrong:** WS connection appears open (`readyState === 1`) but messages stop arriving after the screen locks.
**Why it happens:** iOS Safari suspends JS execution for PWA standalone apps when the screen locks. The WS TCP connection drops silently — no `onclose` event fires while the page is hidden.
**How to avoid:** `visibilitychange` listener checks `readyState` on `visible` event. If closed or closing, reset backoff to 0 and reconnect immediately. Always follow reconnect with `wsCatchUp()`.
**Warning signs:** Fields on one device show stale state after phone wakes from sleep.

### Pitfall 6: Missing `defer tx.Rollback()` in Op Transaction
**What goes wrong:** Failed op inserts leave open transactions that block subsequent writes.
**Why it happens:** Forgetting rollback pattern established in existing handlers.
**How to avoid:** Every transaction in `insertOpAndNotify` must have `defer tx.Rollback(ctx)` before any error-returning call. This is already the pattern in `repository.go`.

### Pitfall 7: Hub Channel Deadlock
**What goes wrong:** Server hangs — all goroutines blocked.
**Why it happens:** If the hub's `broadcast` channel is full and the listener goroutine blocks on `hub.broadcast <- msg`, no other goroutines can send to hub channels either.
**How to avoid:** Make `hub.broadcast` a buffered channel (size 256+). In the hub's broadcast case, use non-blocking sends to individual client `send` channels — drop slow clients rather than blocking.

## Open Questions

1. **Catch-up query scope for `GET /ops/since`**
   - What we know: Returns ops with `lamport_ts > N` for user's accessible entities.
   - What's unclear: "Accessible entities" requires joining `template_assignments` to find which templates the user is assigned to. This could be a complex query if the user has many templates.
   - Recommendation: For Phase 10.1, filter by: `WHERE user_id = $1 OR entity_id IN (SELECT template_id FROM template_assignments WHERE assignee_id = $1)`. Add a covering index if needed.

2. **When `drainQueue()` completes before `wsCatchUp()`**
   - What we know: D-15 says catch-up happens after drain completes.
   - What's unclear: The current `drainQueue()` is async with no completion callback. `wsCatchUp()` needs to run after the queue is fully drained.
   - Recommendation: Chain `wsCatchUp()` at the end of `drainQueue()` — after the `finally` block clears `_draining`. The `window.addEventListener('online', ...)` flow becomes: drain → then catch up.

3. **`emitOp` goroutine context lifetime**
   - What we know: Fire-and-forget goroutines use `context.Background()` with a timeout.
   - What's unclear: If the request context cancels before the goroutine starts, is there a risk of DB connection exhaustion under load?
   - Recommendation: Use `context.WithTimeout(context.Background(), 5*time.Second)` — independent of request context, capped at 5 seconds. This is the same pattern as `cleanupOldDrafts`.
