## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 | 1.5 |
| Label | 12px | 600 | 1.3 |
| Heading | 15px | 600 | 1.2 |
| Display | 26px | 400 | 1.1 |

Notes:
- Body (14px/400): user-entered text fields, list row names, FAQ answers, item descriptions, secondary button labels, external notice text, sub-tab labels (previously 13px — consolidated to 14px)
- Label (12px/600): form field labels (`text-transform: uppercase; letter-spacing: 0.4px`), attribution lines, category headers (`text-transform: uppercase; letter-spacing: 0.5px`), pill text, `.em` secondary metadata (previously 11px — consolidated to 12px)
- Heading (15px/600): card section headers (`.hd h1`), modal titles
- Display (26px/400): wordmark on login.html only — not used in onboarding.html or users.html
- Back-arrow glyph (‹): render at Body size (14px); no separate size declaration needed for a single decorative glyph

**Weight rationale:** Two weights only — 400 (regular) and 600 (semibold). Previously declared 500 and 700 are merged into this pair: all medium/semibold uses map to 600, bold uses map to 600 (the Display wordmark does not need bold distinction in this phase).

**Size consolidation:** 11px → 12px, 13px → 14px, 20px (back-arrow glyph only) → no declaration. Total declared sizes: 4.

**Source:** Extracted from `onboarding.html` lines 17–47, `users.html` lines 17–74, `login.html` lines 19–27. Consolidated per checker feedback.

## Visual Hierarchy

### users.html — Primary Screen (User List Tab)

Focal point: the user list rows. Each row holds the crew member's name (Body/400) as the primary read target, with role pill (Label/600) as the secondary signal. The "Add Crew Member" CTA button is the only Accent-colored element on the page, drawing the eye after the list.

Hierarchy order:
1. Page heading "Team" (Heading/600) — orients the user
2. User rows — name + pill per row (Body/400 + Label/600)
3. "Add Crew Member" button (Accent) — sole action anchor at bottom

### onboarding.html — Primary Screen (My Trainings Tab)

Focal point: the section headers with their completion state (checked or locked). Video links and FAQ items are subordinate to section progress.

Hierarchy order:
1. Section header + progress indicator — tells the trainee what to do next
2. Checklist items within the active (unlocked) section — the immediate work
3. Video link (Accent text) — supplemental, not primary
4. Locked sections (Muted color) — visually recede to reduce cognitive load

## Interaction Contracts

### Auto-Save (onboarding.html — training items)

- Trigger: checkbox tap or video-watched mark
- Behavior: Immediate optimistic UI update → POST `/api/v1/onboarding/saveProgress` in background
- Failure: Show `.field-err` inline error "Couldn't save. Tap to retry." — do NOT revert checkbox state until retry fails twice
- Source: CONTEXT.md D-02

### Invite Link Copy

- Trigger: Tap "Copy Link" button
- Behavior: `navigator.clipboard.writeText(url)` → button text changes to "Copied!" for 2000ms → reverts
- Fallback (if clipboard API unavailable): Select the URL text in a read-only input field

### Nickname Collision (users.html edit form)

- Trigger: "Update Profile" tap
- Behavior: POST to server → 409 response → show `.field-err` under nickname field with collision message — do NOT close form
- Source: CONTEXT.md D-11

### Session Revocation (Force Logout)

- Trigger: Tap "Force Logout" button
- Behavior: `confirm()` dialog with exact copy from destructive confirmation table → on confirm: POST `/api/v1/users/{id}/revoke` → success: close edit form, no badge change needed (lazy 401 on crew side)
- Source: CONTEXT.md D-21, D-22

### Accept Invite (login.html)

- Trigger: Page load with `?token=` in URL
- Behavior: Hide email + password fields → show "Welcome, {name}" heading + set-password form
- Token validation: On form submit — server validates token; on expired/used: show error banner (`.error.show`)
- Source: CONTEXT.md D-16, D-17, D-18

### Sign-Off Form (onboarding.html)

- Trigger: Tap "Sign Off Section" button
- Behavior: Append sign-off form card below the section header (inline, not a modal overlay) → manager fills notes + rating → tap "Confirm Sign-Off" → POST `/api/v1/onboarding/signOff` → replace button with "Signed off" text + attribution line
- Cancel: Tap "Never Mind" → remove the appended form card, restore "Sign Off Section" button
- Validation: Both notes (non-empty) and rating (selection required) must be filled before confirm button is active
- Source: CONTEXT.md D-05, D-06

### Loading States

- Initial page load: Show skeleton rows (2 animated placeholder bars per row) while awaiting first API response
- Pattern: Match Phase 10 skeleton screen pattern (Phase 10 D-24 from CONTEXT.md)
