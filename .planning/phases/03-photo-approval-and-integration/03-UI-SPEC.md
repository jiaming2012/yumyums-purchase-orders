## Spacing Scale

Declared values (inherited from Phases 1 and 2, unchanged):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Chip gaps, badge padding, icon gaps, thumbnail gap from retake link |
| sm | 8px | Card margin-bottom, row gap between items, thumbnail border-radius |
| md | 12px | Default row padding vertical, body horizontal padding, modal inner padding vertical |
| md+ | 16px | Row padding horizontal, card inner padding horizontal, modal inner padding horizontal |
| lg | 24px | Page-level section breaks, modal backdrop vertical centering offset |
| xl | 32px | Empty-state padding vertical |

Exceptions:
- **12px (`md` token):** Not in the standard 8-point set {4, 8, 16, 24, 32, 48, 64}. Approved as a project exception — this is a project-adopted mid-step between 8px and 16px, established as a codebase convention in `purchasing.html` and all existing pages. Retained for visual density on mobile. Same exception carried forward from Phase 1 and Phase 2.
- Photo preview modal: full-screen overlay (width 100vw, height 100vh). Backdrop `rgba(0,0,0,0.85)`. Action buttons (checkmark + retake) centered at bottom with 32px bottom inset.
- Photo thumbnail: 72px × 72px fixed square with `border-radius: 8px` and `object-fit: cover`. Sits inline in the field row after capture is confirmed.
- Touch targets: minimum 44px tall on all tappable rows — inherited from Phase 2 (no change).
- Approval card approve/reject buttons: minimum 44px tall, full-width, stacked vertically with 8px gap.

Source: Phase 2 `02-UI-SPEC.md` spacing scale; 12px extension retained.

## Typography

Inherited from Phases 1 and 2. No new sizes or weights added.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Labels / meta / badges / attribution | 12px | 400 (regular) | 1.4 |
| Body / field label / secondary body / input | 14px | 400 (regular) | 1.4 |
| Heading / card title / checklist name / section header | 15px | 500 (medium) | 1.2 |

Notes:
- **14px/15px gap:** The 1px size difference between body and heading is an inherited characteristic from Phases 1 and 2. Heading distinction is carried primarily by **weight 500**, not size alone. This is intentional — both sizes read at near-identical scale on mobile but the medium weight creates sufficient visual hierarchy without increasing the heading size further.
- Photo thumbnail retake link: 12px, weight 400, `color: var(--info-tx)` — styled as a text link, no underline.
- Approval status badge ("Pending Approval"): 12px, weight 500, `var(--warn-bg)` / `var(--warn-tx)` — reuses the Phase 2 "Requires approval" pill color.
- Approval card meta line (who submitted, when, completion %): 12px, weight 400, `color: var(--mut)`.
- Rejection reason input: 14px, weight 400, `var(--bg)` background — follows `.field input` pattern.
- Approvals tab label: 14px weight 400 (inactive), 14px weight 500 (active) — matches existing `.tabs button` CSS.

Source: Phase 1 `01-UI-SPEC.md` typography contract.

## Interaction Contracts

### Three-Tab Navigation

- **Tabs:** [My Checklists] [Approvals] [Builder] — in this left-to-right order (D-04 from CONTEXT.md).
- **Tab IDs:** `t1` = My Checklists, `t2` = Approvals, `t3` = Builder.
- **Section IDs:** `#s1` = fill-out (Phase 2 runner), `#s2` = approvals (new), `#s3` = builder (previously `#s2` — renumbered).
- **`show(n)` update:** Iterate `[1,2,3]` to hide/show sections and set `.on` state on tab buttons.
- **Approvals tab gate:** Approvals tab button is rendered only if `MOCK_CURRENT_USER.role === 'admin' || MOCK_CURRENT_USER.role === 'manager'`. If not eligible, tab button is not rendered at all (same pattern as Phase 1 builder tab gate, D-11).
- **Default tab:** `t1` (My Checklists) is active on page load. No change from Phase 2.

### Photo Capture — Standalone Field Row

- **Trigger:** Tap anywhere on the photo field row (`.photo-field-row`). The hidden `<input type="file" accept="image/*" capture="environment">` is `.click()`-ed programmatically (D-01).
- **iOS note:** The `<input>` element must be re-created (or reset its `value`) each time to work around the iOS single-use input bug (Pitfall 9 from PITFALLS.md). Create a fresh `<input>` node, append to body, `.click()`, then remove after change event fires.
- **On capture:** `input`'s `change` event fires. Read `e.target.files[0]`. Call `URL.createObjectURL(file)`. Open photo preview modal with the object URL as `<img src>`.
- **Modal appearance:** Full-screen fixed overlay with `rgba(0,0,0,0.85)` backdrop. `<img>` centered, `max-width: 100%`, `max-height: 70vh`. Action bar at bottom: ✓ confirm button (left) and `↻ Retake` (right), 32px bottom inset, 16px gap between buttons.
- **Confirm (✓):** Stores the object URL in `state.responses[templateId].completedItems[fieldId].value` (photo URL string). Closes modal. Renders `.photo-thumb` (72×72px) + "Retake" text link inline in the field row.
- **Retake (↻):** From within the modal — triggers a new `<input type="file">` click (same iOS workaround). Replaces the current preview `<img src>` with the new capture.
- **Retake from confirmed thumbnail:** "Retake" text link tap — reopens the file picker. If user captures a new photo, goes through full preview flow again. If user cancels the picker, the existing thumbnail is preserved.
- **State:** `completedItems[fieldId]` = `{ value: objectURL, answeredBy: CURRENT_USER, answeredAt: ISO string }`.

### Photo Capture — Corrective Action Card

- **Trigger:** Tap the `.photo-stub-btn` button inside a `.fail-card`. The Phase 2 `disabled` attribute and `opacity:0.6` are removed. Same file picker flow as standalone field row.
- **After confirm:** Thumbnail rendered inside the `.fail-card` below the existing note textarea and severity picker. No size change — same 72×72px thumbnail.
- **State:** Stored in `state.responses[templateId].failNotes[fieldId].photo` (object URL string).

### Submit — Approval Routing

- **On "Submit checklist" tap:** Check if `activeTemplate.requires_approval === true`.
  - **If true:** Do NOT set `SUBMITTED_TEMPLATES[tplId] = true`. Instead, set `PENDING_APPROVALS[tplId] = { submittedBy: CURRENT_USER, submittedAt: ISO string, responses: snapshot }`. Show inline confirmation: "Submitted for approval. Waiting for manager review." Button becomes disabled. No fireworks.
  - **If false:** Existing Phase 2 behavior — `SUBMITTED_TEMPLATES[tplId] = true`, fireworks, inline confirm "Checklist submitted. Thanks, {initials}!".
- **Pending approval badge:** On the checklist list card (My Checklists view), if `PENDING_APPROVALS[tplId]` exists, show `.approval-badge` "Pending Approval" beneath the progress bar.

### Approvals Tab — Queue View

- **Entry:** Manager taps Approvals tab.
- **Content:** For each entry in `PENDING_APPROVALS`, render one `.approval-card`:
  1. `.hd`: Checklist name at 15px weight 500.
  2. Meta line: "{submitter name} · {relative time}" at 12px `var(--mut)` (e.g., "Maria G. · Today 2:15p").
  3. Stats row: "{N} of {Y} complete · {F} fail trigger(s) · {P} photo(s)" at 12px `var(--mut)`.
  4. Photo strip (if photos captured): `.photo-strip` horizontal scroll of 72×72px thumbnails.
  5. Action bar: `.btn-approve` + `.btn-reject` stacked vertically with 8px gap.
- **Empty state:** If `PENDING_APPROVALS` has no entries, render `.empty-state`.

### Approve Action

- **Trigger:** Tap `.btn-approve` on an approval card.
- **Behavior (D-06):** One-tap approval. Move `tplId` from `PENDING_APPROVALS` to `APPROVED_SUBMISSIONS`. Call `showToast('Approved ✓')`. Remove the card from the DOM (re-render approvals list). No modal.
- **Toast:** Existing `showToast()` function with message `'Approved ✓'`. 2-second display.

### Reject Action

- **Trigger:** Tap `.btn-reject` on an approval card (D-06).
- **Behavior — step 1:** Reveal rejection reason `<textarea>` below the reject button and `.btn-reject-confirm` below it. The approve button remains visible above.
- **Behavior — step 2:** Manager types reason. Taps `.btn-reject-confirm` "Confirm rejection".
- **Behavior — step 3:** Move `tplId` from `PENDING_APPROVALS` to `REJECTED_SUBMISSIONS` with `{ reason, rejectedBy, rejectedAt }`. Call `showToast('Rejected')`. Remove the card from the DOM. No navigation change (D-07).
- **Cancellation:** If manager taps away from the textarea (blur), the textarea collapses and `.btn-reject-confirm` hides. The reject button returns to its default state. No state mutation occurs.
- **Validation:** `if (!reason.trim()) return;` — do not confirm rejection with empty reason.

### Pre-Built Templates

- **Templates (D-08 + D-09):** Two pre-built templates: "Setup Checklist" and "Closing Checklist".
- **Placement:** Added to `MOCK_TEMPLATES` array alongside any existing templates. The existing templates (Morning Opening Checklist, Closing Checklist) may be replaced or supplemented — executor discretion.
- **Content:** Minimal placeholder items — 2-3 sections with 2-3 fields each. Enough to demonstrate section grouping, at least one photo field, and at least one yes/no field.

### HQ Integration

- **D-10:** Operations tile already links to `workflows.html` in `index.html`. Verify link target is correct. No new UI contract required.
- **sw.js:** Already includes `workflows.html` in ASSETS at cache version v19 (or current). Bump cache version after any new HTML/JS changes in this phase (see sw.js MEMORY note).

### Event Delegation

- `#s2` (approvals tab): ONE `click` listener on `#approvals-body`. Route via `event.target.closest('[data-action]')`.
- `#s1` (fill-out runner): Existing ONE `click` + ONE `input` listeners on `#fillout-body` — extend to handle photo field row taps and corrective action photo button taps.
- Do NOT attach listeners inside render functions.
