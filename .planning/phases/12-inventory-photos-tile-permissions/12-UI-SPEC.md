## Visual Hierarchy (Focal Points)

### inventory.html — History Tab

Primary focal point: the pending review badge count on the tab label (e.g. "History (2)"). This is the first thing a user scans to know whether action is needed.

Visual hierarchy top to bottom:
1. Tab bar — active tab underlined in accent color; pending badge in warn color draws eye first
2. Pending review cards — `.approval-badge "Needs Review"` in warn colors anchors top-right of each card, making them visually distinct from confirmed events
3. Confirmed event cards — subdued, secondary position below pending items
4. Empty state — centered, large heading at 16px weight 600, body at 14px weight 400

No decorative elements. Hierarchy is achieved entirely through color (warn vs muted) and position (pending items first in list).

### index.html — Launcher Grid

Primary focal point: the active tiles (`.tile.active`) with their emoji icons. These are the entry points; placeholder tiles (`.tile.soon`) are visually suppressed.

Visual hierarchy:
1. Active tiles — full color emoji, label at 14px weight 400, full card opacity
2. Soon tiles — muted opacity (0.4), "Soon" badge in `var(--mut)` color, no tap target
3. Page title / wordmark — 22px weight 600 at top, sets context but does not compete with tile grid

Permission-filtered tiles are removed from DOM entirely (not hidden with CSS), so the grid reflows naturally with no visible gaps. No loading skeleton — cached tiles render immediately.

## Interaction Contracts

### Photo Upload Flow (PHOT-01, PHOT-02) — workflows.html

Current state: blob URL stored, no upload. Phase 12 swaps to presigned upload.

```
User taps "Take Photo"
  → Camera input opens (existing <input type="file" accept="image/*" capture="environment">)
  → User captures photo → blob URL created
  → showPhotoPreview(blobURL, onConfirm, onRetake) opens .photo-modal
  → User taps confirm (.photo-confirm-btn)
    → Show "Uploading..." in photo field area
    → POST /api/v1/photos/presign → receive { url, object_key }
    → PUT blobData to presigned URL (direct to DO Spaces)
    → On success: store object_key in field response via autoSaveField()
    → Replace photo area: show .photo-thumb with Spaces URL as src
    → Show "Retake Photo" link below thumbnail
  → User taps Retake Photo
    → Clear current photo from field response
    → Re-open camera input
  → On upload error
    → Show .inline-error "Upload failed. Tap to retry."
    → Retry taps presign + PUT again
```

**Visual states for photo field:**
1. Empty: `.photo-capture-btn` with label "Take Photo"
2. Uploading: button replaced with "Uploading..." text + CSS spinner (inline `border-radius: 50%; animation: spin 1s linear infinite`)
3. Uploaded: `.photo-thumb` (72×72, border-radius 8px) + `.photo-retake-link` "Retake Photo"
4. Error: `.inline-error` with retry

### Receipt Review Queue (INVT-03) — inventory.html

New tab or section within History tab showing pending receipts from Mercury pipeline.

```
User opens inventory.html → History tab
  → Fetch GET /api/v1/purchases/pending
  → If pending items > 0: show badge count on History tab (reuse .sync-badge pattern)
  → Pending items render as .event-card with .approval-badge "Needs Review"
  → User taps a pending item
    → Expand or navigate to review form
    → Pre-filled fields: vendor, date, line items (description, qty, unit, price)
    → If total mismatch: show .correction-banner with mismatch warning
    → If new vendor/item: show inline warning below field
  → User edits fields, taps "Confirm Receipt"
    → POST /api/v1/purchases/confirm { event_id, corrections }
    → On success: item moves to confirmed events list; badge count decrements
    → On error: show .inline-error with retry
  → User taps "Discard Receipt"
    → native confirm() — "Discard this receipt? It will be removed from the review queue."
    → POST /api/v1/purchases/discard { event_id }
```

### Inventory API Swap (INVT-01, INVT-02) — inventory.html

Big-bang mock replacement. All MOCK_ arrays deleted. All tabs fetch from API on load.

```
Page loads
  → Show .skeleton for each tab's list area
  → Fetch GET /api/v1/purchases (History), GET /api/v1/stock (Stock)
  → Replace skeleton with rendered data
  → On error: show .inline-error with retry per section
```

### Tile Permission Filtering (TILE-01) — index.html

```
Page loads → checkAuth() runs (existing)
  → Also fetch GET /api/v1/me/apps
  → Read cached apps list from localStorage['hq_apps']
  → Render tile grid from cached list immediately (prevents flash)
  → When /me/apps resolves:
    → Update localStorage['hq_apps'] with new list
    → For each tile in DOM: if app slug not in response, remove tile element
    → If app slug IS in response: ensure tile shows as .tile.active
  → On fetch error: keep cached tiles; no error shown
```
