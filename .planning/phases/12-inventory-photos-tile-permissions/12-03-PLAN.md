  <action>
    Extend the existing checkAuth() function in index.html to also filter tiles by permissions (per D-16, D-17):

    1. Define tile slug map at top of script block:
    ```javascript
    const TILE_SLUGS = {
      'workflows.html': 'operations',
      'purchasing.html': 'purchasing',
      'users.html': 'users',
      'inventory.html': 'inventory',
      'onboarding.html': 'onboarding',
    };
    const APPS_CACHE_KEY = 'hq_apps';
    ```

    2. Create filterTilesByPermissions(apps) function:
    - Build Set of allowed slugs from apps array
    - For each .tile element in .grid: look up slug via href in TILE_SLUGS map
    - If slug exists but NOT in allowedSlugs set: remove tile from DOM entirely (tile.remove())
    - "Soon" tiles with no href are left as-is (they have no slug mapping)
    - Grid reflows naturally with no gaps

    3. In checkAuth(), after the existing /me fetch:
    - Read cached apps from localStorage[APPS_CACHE_KEY]
    - If cached data exists: call filterTilesByPermissions(JSON.parse(cached)) immediately
    - Fetch GET /api/v1/me/apps
    - On success: update localStorage cache, call filterTilesByPermissions with fresh data
    - On fetch error: keep cached tiles shown (graceful offline degradation)
    - If no cache AND fetch in progress: do nothing special — tiles stay visible until response arrives per UI-SPEC (show all as-is, then filter)

    4. Per Pitfall 5 (first load with no cache): On very first load with no cached data, the tiles will briefly show all active tiles until /me/apps resolves. This is acceptable — it only happens once and the fetch is fast on local network.

    CRITICAL: Do NOT show a loading skeleton or hide all tiles during the fetch. The UI-SPEC says "no loading skeleton — cached tiles render immediately."
  </action>
