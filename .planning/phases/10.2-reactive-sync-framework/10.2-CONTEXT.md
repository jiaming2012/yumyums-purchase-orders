# Phase 10.2: Reactive Sync Framework - Context

**Gathered:** 2026-04-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Extract a shared reactive Store + single write channel from workflows.html. All state mutations flow through `submitOp` → `POST /ops` → server applies business logic + InsertOpAndNotify → entity tables + ops + pg_notify → all clients. Optimistic updates preserved. Store, sync client, and applyOp become a shared `sync.js` module reusable by all tool pages (onboarding, inventory, users).

This phase does NOT add new features — it refactors the sync infrastructure to prevent the class of bugs found in Phase 10.1 verification (components not re-rendering after state changes, stale references, manual render calls).

</domain>

<decisions>
## Implementation Decisions

### File Structure
- **D-01:** Single `sync.js` file (~300 LOC) containing: Store class, LamportClock class, wsConnect/wsCatchUp, submitOp, applyOp, IndexedDB helpers (getDB, idbGet, idbPut, idbDelete, idbGetAll), api() wrapper. Loaded via `<script src="sync.js">` on every tool page.
- **D-02:** Page-specific code (renderers, collections, subscribers) stays inline in each HTML page's `<script>` block. Follows existing `ptr.js` shared-file pattern.

### Migration Scope
- **D-03:** Extract-then-migrate in 3 steps. Step 1: Create sync.js with Store + submitOp + WS + LamportClock + idb. Step 2: Wire workflows.html to sync.js (replace globals with store collections, autoSaveField with submitOp, manual render calls with store.on subscribers). Step 3: Add backend POST /ops endpoint and switch submitOp to call it.
- **D-04:** Each step must have a green test suite before moving to the next. Existing 19 sync tests + 108 workflow tests + 39 persistence tests must all pass at each step boundary.

### Store API
- **D-05:** Pure store with collection-level subscriptions. No computed/derived values in Store — derive helpers are plain JS functions extracted naturally when reused.
- **D-06:** Store API surface:
  - `store.register({ key: initialValue, ... })` — define collections
  - `store.set(key, subKey, value)` — set entry in a key-value collection
  - `store.setAll(key, value)` — replace entire collection
  - `store.get(key)` — read full collection
  - `store.get(key, subKey)` — read single entry
  - `store.delete(key, subKey)` — remove entry
  - `store.on(key, callback)` — subscribe (returns unsubscribe function)
  - `store.off(key, callback)` — explicit unsubscribe
  - `store.batch(fn)` — group mutations, fire subscribers once after fn completes
- **D-07:** Batch is required for: wsCatchUp (applying multiple ops), loadMyChecklists (replacing multiple collections at once), any multi-field update.

### Single Write Channel
- **D-08:** `submitOp(opType, entityId, entityType, payload)` is the single client-side write function. It: saves a pre-op snapshot, applies optimistically to the Store, sends `POST /ops` with client device_id + lamport_ts, handles 409 conflict (revert to winner), handles network error (rollback to snapshot).
- **D-09:** Client-originated ops include real device_id and lamport_ts from LamportClock. This replaces the transitional `device_id: "server"` + `_recentSaves` hack — proper self-echo suppression via `op.device_id === LAMPORT_CLOCK.deviceId`.
- **D-10:** `autoSaveField` is replaced by `submitOp` with a per-field debounce wrapper. Same 400ms debounce behavior, but the debounce is outside submitOp (submitOp itself is immediate).

### Backend POST /ops
- **D-11:** `POST /ops` is the primary write path. Server receives `{ op_type, entity_id, entity_type, payload, lamport_ts, device_id }`, routes by op_type to existing business logic (saveResponse, submitChecklist, approve, reject, create/update/archive template), then calls InsertOpAndNotify.
- **D-12:** Old endpoints (`saveResponse`, `submitChecklist`, etc.) become thin wrappers that construct an OpInput and delegate to the OpHandler internally. Kept for backward compatibility during migration.
- **D-13:** Tech debt: Remove old endpoints entirely once all clients use POST /ops (Option A). Not in this phase's scope.
- **D-14:** Business logic (fail note validation, idempotency keys, template archiving checks) is preserved inside the op handler's routing — not stripped out.

### Subscriber Patterns
- **D-15:** When the runner view is open, subscribers re-render the runner. When the list view is open, subscribers re-render the list. The Store doesn't know about views — subscribers check `fillState.view` or equivalent to decide what to render.
- **D-16:** `loadMyChecklists` becomes `store.refresh('myChecklists')` or equivalent — always fetches data from API (never early-returns based on DOM state), replaces collections in a batch, subscribers fire automatically.

### Tech Debt (not in this phase)
- Fine-grained key-level subscriptions (Option A from discussion) — upgrade path if collection-level proves too coarse
- Remove old write endpoints entirely (keep only POST /ops)

### Claude's Discretion
- Store class internal implementation (Map vs plain object, notification scheduling)
- IndexedDB version bump strategy (if syncMeta store changes shape)
- Whether api() wrapper stays in sync.js or becomes a separate concern
- SW rebuild via `node build-sw.js` after sync.js is created (Workbox must precache it)
- Test organization for sync.js unit tests vs integration tests

</decisions>

<specifics>
## Specific Ideas

- **submitOp replaces autoSaveField** — the debounce wrapper around submitOp should preserve the 400ms debounce and "Saving..." status indicator behavior. The debounce is per-field (same SAVE_DEBOUNCE pattern), but the actual write goes through submitOp → POST /ops.
- **store.batch for data refreshes** — when loadMyChecklists-equivalent refreshes all data from the API, it wraps store.setAll calls in a batch so subscribers fire once, not per-collection.
- **Self-echo via device_id** — once the client sends ops with its real device_id, `applyOp` can use the existing `op.device_id === LAMPORT_CLOCK.deviceId` check. The `_recentSaves` hack and the server's `device_id: "server"` pattern both go away.

</specifics>
