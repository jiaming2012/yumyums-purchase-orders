## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Tab pill gaps, day-chip gaps, icon-to-label gap in back link |
| sm | 8px | Card margin-bottom, field margin-bottom, intra-card item gaps |
| md | 16px | Default horizontal padding (`.hd`, `.row`, `.cat`), default element spacing |
| lg | 24px | Page-level top padding, major list section breaks |
| xl | 32px | Not used directly — composed from sm+lg as needed |
| 2xl | 48px | Not used directly in existing pages |
| 3xl | 64px | Not used directly in existing pages |

Exceptions:
- Touch targets: `min-height: 44px` on interactive rows and buttons (iOS HIG requirement — D-13 items must be freely tappable)
- Tab button padding: 12px top/bottom (nearest multiple of 4 to the project's 10px pattern — standardized for this phase)
- Body safe-area padding: `env(safe-area-inset-top) 12px env(safe-area-inset-bottom)` — inherited from project PWA pattern
- Progress bar height: 4px (checklist list view) and 8px (checklist runner view) — multiples of 4

**Source:** Direct inspection of workflows.html lines 17-88

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body / item label | 14px | 500 (medium) | 1.4 |
| Meta / muted subtext | 12px | 400 (regular) | 1.4 |
| Category header / section header | 12px | 500 (medium) | 1.0 — uppercase with letter-spacing 0.5px |
| Tab button | 14px | 500 (medium) | 1.0 |

Notes:
- **Scale rationale:** The existing project uses a tight 12/13/14/15px range across all pages. For this phase the scale is consolidated to 12px / 14px (two distinct sizes) to avoid the sub-3px ambiguity flagged by the checker. The 13px tab button size from other pages is bumped to 14px to align with Body role. The 15px `.hd h1` heading pattern is also aligned to 14px weight 600 for checklist runner title and hire card header — this is a deliberate simplification for this phase only.
- No display/heading size needed — this is a list-and-card UI with no marketing copy
- Field labels (e.g., form inputs in Builder) use 12px weight 500 uppercase — not needed in Onboarding (no input forms)
- Two weights only: 400 (regular) for meta subtext, 500 (medium) for all other roles

**Source:** Direct inspection of workflows.html CSS lines 19-67; scale consolidated per checker Dimension 4 guidance

## Focal Points

Declared per primary screen to anchor visual hierarchy:

### My Trainings list view
**Focal point:** The checklist card's title text (14px weight 500, `--txt`) in the top-left of each card. The progress bar (4px, `#27ae60` fill) immediately below the title serves as a secondary anchor — color and position draw the eye to completion state. Everything else (meta subtext, progress count) is subordinate in muted text.

**Hierarchy order:** Card title → progress bar fill → "X of Y tasks" count → section locked/open indicator

### Checklist runner view (open checklist)
**Focal point:** The active section header (12px uppercase, weight 500, `--txt` on `--bg`) with its expand chevron. The section header spans full width and is the only interactive affordance when a section is collapsed — it is the entry gate for all content below. Within an expanded section, the checkbox column anchors each item row on the left.

**Hierarchy order:** Section header (active/locked state) → item checkboxes (left-aligned column) → item label text → attribution subtext → "Request Sign-Off" CTA (bottom of section, full width, accent color)

## Interaction Contracts

### Sequential Section Lock
- Locked sections render header with "🔒 Locked" label + no expand affordance
- Active sections render with expand/collapse chevron on header
- Tapping a locked section header: no action, no state change
- Tapping an active section header: toggles expand/collapse (CSS display toggle, no animation library)
- Items within an active section are tappable in any order (D-13)

### Checkbox Check / Uncheck
- Tap toggles checked state immediately
- Re-render section item list after state mutation
- Checked state: `background: rgba(39,174,96,0.12); border-color: #27ae60` with "✓" glyph
- Unchecked state: `border: 1.5px solid var(--brd); background: transparent`
- Attribution shown on checked items: "Checked by [name]" in 11px `var(--mut)` below label

### Video Part "Watch Video" link
- Tap opens URL via `window.open(url, '_blank')` — never an `<iframe>` embed
- Checkbox below part title is independent of the link tap — crew manually checks after watching
- "Watch Video" button: plain text link styled with `color: var(--info-tx); font-size: 14px; font-weight: 500`

### FAQ Expand Gate
- FAQ section renders with "View Required" badge on section header when `FAQ_VIEWED[hireId]` is falsy
- First tap on FAQ section header: sets `FAQ_VIEWED[hireId] = true`, re-renders section header badge to "✓ Viewed"
- Q&A pairs accordion: subsequent taps toggle individual Q&A items open/closed
- FAQ viewed state is a prerequisite for checklist completion (Pitfall 5 from RESEARCH.md)

### Sign-Off Request (Crew)
- "Request Sign-Off" button appears at bottom of section when all items are checked and `requiresSignOff: true`
- Tap mutates `SECTION_STATES[hireId][secId]` to `'pending_signoff'` and re-renders
- Section shows "Pending Sign-Off" badge; items become read-only (no checkboxes interactive)
- Button label: "Request Sign-Off" (not "Submit" — avoids confusion with checklist submission)

### Sign-Off Approve/Send Back (Manager)
- Manager reads checklist in read-only mode (no `data-action` on checkboxes — Pitfall 4 from RESEARCH.md)
- Pending sections show "Approve" (green) + "Send Back" (red) buttons at section bottom
- "Approve" tap: `SECTION_STATES[hireId][secId] = 'signed_off'`, unlocks next section, re-renders
- "Send Back" tap: `SECTION_STATES[hireId][secId] = 'active'`, section re-opens for crew edits, re-renders

### Assign Additional Training (Manager)
- "+ Assign Training" button below hire's card in Manager Active view
- Tap opens inline picker (not a modal) — a `.card` that expands below the button listing available templates as rows
- Tap a template row to add to hire's `assignedTemplates`, re-render hire card
- No confirmation required (low-risk additive action; crew will see new checklist in My Trainings)

### Drill-Down Navigation (My Trainings list → Checklist runner)
- Tap checklist card opens runner view (state mutation, no page navigation)
- Back button (`‹ My Trainings`) appears at top of runner — same `.back` class as all existing pages
- Back tap returns to list view (state mutation, re-render)

## Accessibility Minimums

- All interactive elements: `min-height: 44px` (iOS HIG touch target)
- Tappable cards: `cursor: pointer` and `touch-action: manipulation` (inherited from `*` reset)
- Locked section headers: no interactive affordance rendered (no pointer cursor, no tap highlight)
- Color is never the sole indicator of state — badges include text labels alongside color
