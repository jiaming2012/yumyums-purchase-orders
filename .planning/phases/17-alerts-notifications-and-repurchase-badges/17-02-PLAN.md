    **scheduler.go — Cutoff reminder (ALRT-01, per D-08, D-09, D-10):**
    - Add helper `getAlertRecipients(ctx, pool, roleFilter string) []AlertRecipient` that queries users with the given role in their roles array, returning AlertRecipient with notification_pref from the new column. Use this for all alert triggers.
    - In `runCutoffCheck`, after computing `cutoffCandidate` and before the lock check:
      1. Compute `reminderTime := cutoffCandidate.Add(-24 * time.Hour)`
      2. Check if `now` is within a 15-minute window around reminderTime: `now.After(reminderTime.Add(-15*time.Minute)) && now.Before(reminderTime.Add(15*time.Minute))`
      3. Check deduplication: if `config.ReminderSentWeek` equals the current week_start string, skip
      4. If both conditions pass, call `sendCutoffReminder(ctx, pool, cutoffCandidate, config)`
    - `sendCutoffReminder(ctx, pool, cutoffCandidate, config)`:
      1. Count current PO items: `SELECT COUNT(*) FROM purchase_order_items poi JOIN purchase_orders po ON po.id = poi.po_id WHERE po.status = 'draft'`
      2. Get recipients with "order" permission via getAlertRecipients (per D-09 — users who can add to PO)
      3. Build message per D-10: "Cutoff Reminder: {weekLabel} order has {N} items. Cutoff is {DayName} at {HH:MM} {TZ}."
      4. Call `EnqueueAlert(AlertJob{Recipients: recipients, Subject: "Cutoff Reminder", Body: message})`
      5. Update cutoff_config: `UPDATE cutoff_config SET reminder_sent_week = $1 WHERE id = $2` with current week_start string

    **scheduler.go — Low-stock alerts (ALRT-02):**
    - Add `runLowStockCheck(ctx, pool)` called from the same ticker loop (after runCutoffCheck):
      1. Query stock items with level='low' using inventory.ClassifyStockLevel (imported from inventory package, already a shared function per quick task 260422-not)
      2. Compute current week_start string (Monday-based or matching PO convention)
      3. For each low item, INSERT INTO low_stock_alert_log (item_description, week_start) ON CONFLICT DO NOTHING — collect only newly-inserted descriptions
      4. If any new low items found, get admin recipients via getAlertRecipients(ctx, pool, "admin")
      5. Build message: "Low Stock Alert: {N} items below threshold: {item1}, {item2}, ..."
      6. EnqueueAlert with subject "Low Stock Alert"

    **scheduler.go — Badge reset check (REP-02):**
    - Add `runBadgeResetCheck(ctx, pool)` called from ticker loop:
      1. Query badge_reset_config (single row)
      2. If no row exists, return (unconfigured)
      3. Compute next reset time from day_of_week + reset_time + timezone (same DST-safe pattern as cutoff)
      4. If now is past the reset time and last_reset_at is before the reset time: `UPDATE badge_reset_config SET last_reset_at = now()`
      5. This effectively "resets" badges because GetStockHandler (Plan 03) filters repurchase_events by last_reset_at

    **service.go — Shopping completion alert (ALRT-06, per D-11, D-12, D-13):**
    - Replace the NotifyVendorComplete stub with real implementation:
      1. Query the shopping list with all vendor sections and items: get vendor_name, item_name, quantity, checked status for each item in the list
      2. Identify missing items (checked = false)
      3. Get the completing user's info (the userID passed to CompleteVendorSection)
      4. Build recipient list per D-12: admins + the shopper who completed. Query users with notification_pref.
      5. Build message per D-13: "Shopping Complete: {vendor_name} — Completed by {user_name}. Missing items: {item1} (qty {N}), {item2} (qty {N})" or "All items checked" if none missing
      6. Call EnqueueAlert (non-blocking per Pitfall 2 — this runs before tx.Commit but enqueue is instant)
    - IMPORTANT: Also pass `userID` to NotifyVendorComplete (update function signature) so the shopper can be identified. Update the call site in CompleteVendorSection accordingly.
    - NotifyVendorComplete should also INSERT repurchase_events for checked items (for Plan 03 REP-01). For each shopping_list_item WHERE checked = true AND purchase_item_id IS NOT NULL, INSERT INTO repurchase_events (shopping_list_id, purchase_item_id, quantity). Use the pool (not tx) since the stub already receives pool. Note: only items with purchase_item_id get badges (per Research open question 4).
