# Phase 17 Plan 01: Alert Infrastructure, Notification Preferences, and Repurchase Badge API

## Objective

Implement the backend foundation for async alerts and repurchase tracking:
- DB migrations for notification preferences on users + repurchase tracking on shopping list items
- Alert delivery infrastructure: Zoho Cliq webhook sender + email (SMTP) sender, async queue
- Wire shopping completion alert to replace NotifyVendorComplete stub
- Extend cutoff scheduler with 24-hour reminder check
- Extend GET /api/v1/inventory/stock response with repurchase badge data
- Add GET/PUT /api/v1/users/:id/notification-preference API

## Context

@.planning/phases/17-alerts-notifications-and-repurchase-badges/17-CONTEXT.md

## Tasks

### Task 1: DB migrations — notification_channel on users + repurchase tracking

**Type:** auto
**Done when:**
- `0041_notification_channel.sql` adds `notification_channel TEXT NOT NULL DEFAULT 'zoho_cliq'` to users table with CHECK constraint `('zoho_cliq', 'email')`
- `0042_repurchase_tracking.sql` creates `repurchase_log` table: `id`, `purchase_item_id`, `shopping_list_id`, `quantity`, `repurchased_at`; plus `repurchase_reset_config` single-row table with `day_of_week`, `reset_time`, `timezone`, `last_reset_at`
- Both migrations follow goose BEGIN/COMMIT pattern and have Down sections

**Implementation:**
1. Create `0041_notification_channel.sql` — ALTER TABLE users ADD COLUMN notification_channel
2. Create `0042_repurchase_tracking.sql` — CREATE TABLE repurchase_log and repurchase_reset_config

### Task 2: Alert infrastructure — Zoho Cliq sender + email sender + async queue

**Type:** auto
**Done when:**
- `backend/internal/alerts/` package exists with:
  - `types.go` — Alert struct (type, payload fields: message, recipient email, channel)
  - `sender.go` — `SendZohoCliq(webhookURL, message string) error` and `SendEmail(smtpAddr, from, to, subject, body string) error`
  - `queue.go` — `Queue` struct with unbuffered channel, `Start(ctx)` goroutine, `Enqueue(alert Alert)` non-blocking send
  - `config.go` — `Config` struct loaded from env vars: `ZOHO_CLIQ_WEBHOOK_URL`, `SMTP_ADDR`, `SMTP_FROM`
- `SendZohoCliq` sends `{"text": message}` JSON POST to webhook URL; gracefully no-ops if webhook URL is empty
- `SendEmail` uses `net/smtp`; gracefully no-ops if SMTP_ADDR is empty
- Queue goroutine dispatches via user's notification_channel (zoho_cliq or email)

**Implementation:**
1. Create `alerts/types.go` with Alert struct
2. Create `alerts/sender.go` with Zoho Cliq HTTP sender and SMTP email sender
3. Create `alerts/queue.go` with async dispatch goroutine
4. Create `alerts/config.go` with env-var-based Config

### Task 3: Wire alerts into shopping completion and cutoff scheduler

**Type:** auto
**Done when:**
- `NotifyVendorComplete` in `purchasing/service.go` is replaced with real alert dispatch:
  - Queries missing items (unchecked) for the vendor section's shopping list
  - Gets admin users + assigned shopper contact info
  - Enqueues `CompletionAlert` with vendor name, missing items, who completed it
- `purchasing/scheduler.go` extended with `runReminderCheck`:
  - Called on each 15m tick (after cutoff check)
  - If cutoff is configured and 24h reminder has NOT been sent this week, sends reminder
  - Reminder tracks sent state via `alert_log` table or a simple in-memory flag per week_start
  - Reminder message includes: week label, current item count, cutoff day/time
- Alert queue `Start` called from `StartScheduler` (single startup point for background goroutines)

**Implementation:**
1. Add `alert_log` table in a new migration `0043_alert_log.sql` (type, week_start, sent_at)
2. Replace NotifyVendorComplete stub with real alert dispatch
3. Add runReminderCheck to scheduler.go
4. Wire alerts.Queue into purchasing package via constructor injection

### Task 4: Repurchase badge — record on completion + extend stock API

**Type:** auto
**Done when:**
- `CompleteVendorSection` in `purchasing/service.go` writes checked items to `repurchase_log` (purchase_item_id, shopping_list_id, quantity, repurchased_at) after marking section complete
- `inventory/handler.go` GetStockHandler response includes `repurchase_badge *RepurchaseBadge` per stock item:
  - Badge present when repurchase_log has entry for that item since last reset (or ever if no reset configured)
  - Badge struct: `{qty: int, repurchased_at: time.Time}`
- `purchasing/service.go` or new `purchasing/repurchase.go` has `RecordRepurchase(ctx, pool, sectionID) error` function
- New route `POST /api/v1/purchasing/repurchase-reset` allows admin to trigger reset (sets last_reset_at on repurchase_reset_config)

**Implementation:**
1. Add RecordRepurchase function that inserts checked items into repurchase_log
2. Call RecordRepurchase in CompleteVendorSection (within transaction)
3. Extend GetStockHandler to LEFT JOIN repurchase_log and include badge data
4. Add repurchase reset route for admin

### Task 5: Notification preference API

**Type:** auto
**Done when:**
- `GET /api/v1/users/:id/notification-preference` returns `{"notification_channel": "zoho_cliq"}` (admin or self)
- `PUT /api/v1/users/:id/notification-preference` accepts `{"notification_channel": "zoho_cliq|email"}`, validates at least one valid value, updates users table
- Routes registered in server/main.go under existing `/api/v1/users/` router group
- Auth: admin can edit any user; user can edit their own preference

**Implementation:**
1. Add GetNotificationPreference + UpdateNotificationPreference functions to users/db.go
2. Add handlers to users/handler.go (if it exists, else create)
3. Register routes in main.go
