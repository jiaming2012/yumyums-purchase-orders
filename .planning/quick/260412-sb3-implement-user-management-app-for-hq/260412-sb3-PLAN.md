**docs/user-management-api.md** — Create backend design document with:

Data model (4 tables):
- `users` — id (UUID), email (unique), password_hash (nullable for invited), role (enum: superadmin/admin/manager/team_member), display_name, status (enum: invited/active), invited_at, accepted_at, created_at
- `hq_apps` — id (UUID), name, icon (emoji), enabled (bool)
- `app_permissions` — id, app_id (FK), role (nullable), user_id (nullable FK), CHECK constraint (role XOR user_id)
- `sessions` — id, token_hash, user_id (FK), expires_at, created_at

Superadmin config section:
- `config/superadmins.yaml` format: list of emails
- Bootstrapped on server startup, cannot be assigned/revoked via API
- Shows CONFIG badge in UI, edit disabled

API contracts (REST, all JSON):
- POST /api/auth/login {email, password} -> {user, token} or 401
- POST /api/auth/logout (session token in header) -> 204
- GET /api/users (admin+) -> [{id, email, display_name, role, status}]
- POST /api/users/invite {email, role} (admin+) -> 201 {user with status:invited}
- PATCH /api/users/:id {role, display_name} (admin+) -> 200 {user}
- DELETE /api/users/:id (admin+) -> 204
- POST /api/users/:id/reset-password (anyone with Users app access) -> 204 (sends reset email)
- POST /api/invites/accept {token, password} -> 200 {user with status:active}
- GET /api/me/apps -> [{id, name, icon}] (drives HQ tile filtering)
- GET /api/apps/permissions (admin+) -> [{app, role_grants, user_grants}]
- PUT /api/apps/:id/permissions {role_grants, user_grants} (admin+) -> 200

PWA integration plan:
- How index.html will call GET /api/me/apps to filter visible tiles
- Graceful fallback when offline (show all tiles, auth on navigation)

Invitation flow detail:
- Admin sends invite -> user row created with status:invited, password_hash null
- Email sent with acceptance link containing one-time token
- User clicks link -> sets password -> status becomes active
